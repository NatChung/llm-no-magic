(?) What is a System prompt

The hidden instruction that's silently prefilled before your message when you chat with ChatGPT / Claude. For example: "You are a friendly support agent, answer in bullets, max 3 points."

The "product layer" of a SaaS product is exactly this — you type "help me", the product quietly prepends instructions before sending to the model.

This tab lets you see: how much the model's output changes with vs. without a system prompt.

(?) Raw prompt vs Chat template — what's the difference

Raw prompt: feeds your text directly to the model for continuation. The model has no idea who's speaking or who's asking whom (same as Tab ①).

Product layer (chat template): wraps your text with Qwen3's agreed markers before sending. The markers look like this:

<|im_start|>system
You are X
<|im_end|>
<|im_start|>user
help me
<|im_end|>

This tells the model "this part is system, this part is user". Expand the "Final prompt actually sent to the model" preview below to see exactly how the markers wrap it.

(?) But what IS <|im_start|> exactly?

It's not a string — it's a single token.

Looks like 12 characters, but to the model it's 1 token (Qwen3 vocab id 151644). The tokenizer recognises it atomically — won't split into </|/i/m/...

"<|im_start|>"  ← what you see (12 chars)
[151644]        ← what the model sees (1 token id)

Lives in the same vocab list as 一 / 年 / 月 — entry #151644.

Why does this token exist?

The model essentially only does "continue the token sequence" — give it text, it predicts the next token.

Problem: feed it 一年有幾個月? — how does it know to answer, vs essay-continue (like "一年有幾個月? 答案是...")?

<|im_start|> / <|im_end|> are role-boundary markers:

<|im_start|>user        ← "user starts speaking"
一年有幾個月?
<|im_end|>              ← "user is done"

<|im_start|>assistant   ← "now you (assistant) speak"

During training, the model saw billions of conversations in this format and learned: "after <|im_start|>assistant\n, generate a reply, stop at <|im_end|>".

一年有幾個月?<|im_end|>

        (?) What is Thinking mode / the <think> block
        

          
Reasoning models like Qwen3 can "think in their head" before answering. They emit reasoning tokens wrapped in <think>...</think>, and only then emit the answer.

          
    
            
  • Direct answer (skip thinking): the client force-injects an empty <think></think>, so the model has no room to think → it answers directly. Fast, but easy to get wrong on hard problems.
    • 
            
  • With thinking: leaves room for the model to think → you see it actually write the reasoning out as tokens. Slower, but usually correct.
    • 
          

          
The apple problem (Dad has 3 apples, son has 2 more than him — how many does the son have?) is designed to show this difference — direct answers often miscalculate, thinking gets it right.

        (?) What are Agent / Tool / <tool_call> / Turn
        

          

            
Agent: a chatbot answers what you ask. An agent calls tools on its own (read files, write files, run shell...) to get things done, then answers. Underneath it's still token prediction — just with a trigger pattern the client knows how to recognize.

          

          

            
Tool: a ready-made Python function (this repo ships 4: get_time / read_file / write_file / exec_bash) plus a JSON schema describing it.

          

          

            
<tool_call> ... </tool_call>: Qwen3's agreed XML tag. When the model wants to call a tool, it emits this string:

            
<tool_call>
{"name": "get_time", "arguments": {}}
</tool_call>

            
The backend sees it, parses the JSON, calls the matching Python function, then injects the result (e.g. a timestamp) back into the conversation and feeds it to the model again.

          

          

            
Turn: one cycle of the multi-turn loop. Turn 1 = model emits text for the first time (possibly a tool_call). Turn 2 = model sees Turn 1's tool result and emits text again. Repeat until the model stops emitting tool_call → final answer.

⑤ Commands, Scripts & APIs

        
One chapter to cover "how your computer gets things done."

        
Computers aren't black magic. They only understand commands you type, scripts that bundle commands together, and APIs that let your computer ask other computers for what it can't do itself. Once you see these three layers, Skill / MCP follow naturally.

      


      

        
§1. Commands = how you talk to a computer

        
Computers offer two interfaces:

        
    
          
  • Graphical (GUI): click icons, pick menus, drag files
    • 
          
  • Commands (CLI): type words telling the computer what to do
    • 
        

        
You already have a CLI: macOS has Terminal (in Applications / Utilities), Windows has PowerShell or cmd, Linux has shell. Open one — black background, white text, blinking cursor waiting for you to type.

        
Computers ship with a bunch of built-in commands (built into macOS / Linux / Windows):

        
# Built-in commands (ship with the OS)
ls                    # list what's in this folder
cd Downloads          # change to Downloads folder
mkdir photos          # make a new folder called photos
mv *.jpg photos/      # move all .jpg files into photos/

        
Think of these as "the computer's native vocabulary." Built-ins are limited though — for anything special (compressing video, scraping data, processing PDFs) you install extensions:

        
# Install new capabilities
brew install ffmpeg   # macOS: video processing tools
brew install yt-dlp   # YouTube downloader
brew install jq       # JSON processing

        
Now the computer knows a few more "words." ffmpeg is callable.

        

          Plain English: Commands = the computer's vocabulary. Default set, extensible.
        

      


      

        
§2. Scripts = bundling commands together

        
A single command does one thing. Real tasks usually need many steps.

        
Example: "organize ~/Downloads":

        
    
          
  1. list every file
    2. 
          
  2. decide based on the extension
    3. 
          
  3. for .pdf: move to pdfs/
    4. 
          
  4. for .png / .jpg: move to images/
    5. 
          
  5. for anything else: move to misc/
    6. 
        

        
This needs two things:

        
    
          
  • Repetition (do this once per file)
    • 
          
  • Decisions (different extension → different destination)
    • 
        

        
Bundle multiple commands into a file, add conditionals and loops, and you've got a script.

        
#!/bin/bash
# organize.sh — tidy ~/Downloads

for file in ~/Downloads/*; do
  case "${file##*.}" in
    pdf)        mv "$file" ~/Downloads/pdfs/ ;;
    png|jpg)    mv "$file" ~/Downloads/images/ ;;
    *)          mv "$file" ~/Downloads/misc/ ;;
  esac
done

        
    
          
  • for = repeat (run once per file)
    • 
          
  • case = decide (different action per extension)
    • 
          
  • mv = a built-in command
    • 
        

        
Don't worry about syntax — the point is seeing for (repetition) + case (decision). That's a script: a chain of commands + control flow.

        
Repetition + decisions + sequence together are called control flow. That's the power scripts add over single commands.

        
Save as organize.sh, run bash organize.sh — the computer tidies 30 files while you sip water.

        

          
Plain English: Script = a chain of commands plus conditionals and loops, saved as a file, run with one click.

          
2026 reality: You probably won't Google bash syntax yourself — you'll have AI write it for you. But you need to read this layer — otherwise you won't catch AI mistakes.

        

      


      

        
§3. APIs = hiring another computer

        
Problem: your computer isn't omnipotent.

        
Analogy time. In real life, you aren't omnipotent either — you don't pull your own teeth when they ache, you find a dentist. Why?

        
    
          
  • The dentist has different knowledge in their head (bite, nerve maps)
    • 
          
  • Different skills in their hands (drill, anaesthesia)
    • 
        

        
Same in the computer world. Every computer has different knowledge + different skills. For example:

        

          
            

              Computer
              Knowledge in head
              Skills in hand
            

          
          
            
Database host10 years of company ordersLightning-fast queries

            
Weather hostA network of sensorsForecast weather

            
Payments host (Stripe)Financial regulations, payment specsCharge credit cards

            
AI hostA trained large modelGenerate text, translate

          
        

        
Your laptop can't do these — data too big, equipment too specialised, requires licences, requires massive compute.

        
So what to do? Same answer as real life: ask someone else.

        
The way one computer asks another is called an API (Application Programming Interface):

        
# Ask the weather host: what's the weather in Taipei?
curl "https://api.weatherapi.com/v1/current.json?key=YOUR_KEY&q=Taipei"

# Response:
{"current":{"temp_c": 28, "condition":{"text": "Sunny"}}}

        
curl is a built-in command that sends a network request. The weather host receives it, looks up its sensor network, returns JSON.

        
Almost every cloud service (maps, translation, search, payments, email) exposes an API. APIs were designed to be called by programs — hence "Programming" in the name.

        

          Plain English: API = the standard format for your computer to ask another computer (over the network) for help.

⑦ MCP

        
When tools aren't fixed in advance, the AI picks them at runtime.

      


      

        
§1. Recap — what's still missing

        
By chapter ⑥ you know:

        
    
          
  • Commands / scripts / APIs (⑤) — how to make a computer do things
    • 
          
  • Skills (⑥) — how to bundle the above into selectable capabilities the AI loads on demand
    • 
        

        
Both assume one thing: tools are fixed, decided up front. You wrote weather.py into a skill, so the AI can only check weather that way.

        
So what if tools aren't fixed?

      


      

        
§2. Main scenario: controlling web pages

        
Example: "grab the price of this product on eBay."

        
Write a script? Amazon's HTML structure is nothing like eBay's — a script works for one site only.

        
Use APIs? Some sites have one, many don't. You can't keep writing wrappers forever.

        
Worse: sites redesign regularly. The script that works today may break next week.

        
So how does AI handle this?

        
It just "looks at what's there": read the current page, then decide which button to click or which text to grab.

        
But the AI can't see the page on its own — something has to look and act on its behalf. That's the problem MCP solves.

      


      

        
§3. What MCP is

        
MCP = Model Context Protocol. Anthropic published it in November 2024 as an open spec — anyone can implement it.

        
What it solves: AI is too tightly bound to a fixed set of tools.

        
Analogy: USB-C for AI applications (Anthropic's own framing).

        

          
            

              Scene
              Before
              Now
            

          
          
            
Charging cablesProprietary plugs / mini-USB / micro-USB / Lightning...One USB-C cable for every phone and tablet

            
AI ↔ toolsEvery AI app writes its own integrations (Claude one way, ChatGPT another)One MCP protocol — any AI connects to any tool

          
        

        
How it runs: a client (the AI side) and a server (the tool side) talk over JSON-RPC. The client asks the server what tools it offers, then calls them on demand. Full trace in §4.

        
The key point: tools aren't hard-coded — they're queried at runtime. Swap servers, get a new toolset, without changing a line of AI-side code.

      


      

        
§4. A real trace

        
Using Playwright MCP server (a Microsoft open-source server purpose-built to let AI drive a browser) for a simple task.

        
Task: "screenshot example.com for me."

        
→ tools/list                            ← AI starts up, asks the server: what can you do?
← [browser_navigate, browser_click, browser_screenshot, ...]
                                        ← server lists what it offers

→ browser_navigate(url="example.com")   ← AI picks "open page"
← {success: true, title: "Example Domain"}
                                        ← server really opens the browser

→ browser_screenshot()                  ← AI picks "screenshot"
← {image: "data:image/png;base64,..."}  ← returns the screenshot bytes

→ "Screenshot of example.com is above" ← AI wraps up the reply

        
Things to notice:

        
    
          
  • The AI doesn't know the tools up front — it asks at startup
    • 
          
  • The toolset changes when you change servers — want database access? Switch to a DB MCP server, the tools become db_query / db_insert
    • 
          
  • The AI-side code doesn't change — only the partner changes
    • 
          
  • Installing an MCP server is granting it the matching permission — Playwright MCP can drive your browser, Filesystem MCP can read your files. Read what each server does and decide whether you trust it (community-maintained servers carry different trust than ones you wrote yourself)
    • 
        

        
That's the payoff of "one protocol, every tool".

§3. Git repo: a folder that remembers

        
The simplest answer: a git repo.

        
Git repo = an ordinary folder + memory.

        
Analogy: you've used Google Docs, right? Google Docs has "Version history" (File → Version history) — it auto-saves snapshots and you can roll back to any past version.

        
Git repo applies the same concept to a whole folder:

        
    
          
  • You save the "current state" (called a commit)
    • 
          
  • After changes, you can diff, you can roll back
    • 
          
  • Push the folder to the cloud (GitHub) so another computer can download it
    • 
        

        
Why does this matter for non-engineers?

        
    
          
  • AI edited a file wrong → you can reset to the previous commit
    • 
          
  • AI helped you build a useful SKILL.md → push it to GitHub, others can use it too
    • 
          
  • You built a project; six months later you want to see how → check git history
    • 
        

        
Three key words:

        
    
          
  • repo = the folder that remembers
    • 
          
  • commit = "save the current state as a snapshot"
    • 
          
  • push = "sync the local copy up to GitHub"
    • 
        

        
Syntax details don't matter (you can't learn them fast enough on your own); the point is below.

      


      

        
§4. How to use it with AI — you don't type commands

        
In practice, you talk to the AI and the AI runs git for you. Claude Code, for example, works inside a folder — open a repo and Claude Code treats it as the workspace, committing for you.

        
A simple trace:

        
You → "Tidy up the old drafts in this repo and commit them"

AI  → ls drafts/                   (list current files)
    → mv drafts/old-*.md archive/  (move to archive)
    → git add archive/ drafts/
    → git commit -m "archive old drafts"
    → "Done. Moved 5 old drafts to archive/, committed it. Push too?"

You → "Push"
AI  → git push
    → "Pushed to GitHub."

        
What you read is the AI's conversation, not git commands. You just need to follow what AI is doing and confirm it looks right.

        
The upside:

        
    
          
  • You don't have to memorise git syntax
    • 
          
  • AI screws up (wrong commit, broken file) — you say "reset to the previous commit" and AI does it
    • 
          
  • The whole repo is the shared workbench between you and AI — SKILL.md, scripts, conversation logs all sit here
    •
