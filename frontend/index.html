(?) What is a System prompt

The hidden instruction that's silently prefilled before your message when you chat with ChatGPT / Claude. For example: "You are a friendly support agent, answer in bullets, max 3 points."

The "product layer" of a SaaS product is exactly this — you type "help me", the product quietly prepends instructions before sending to the model.

This tab lets you see: how much the model's output changes with vs. without a system prompt.

(?) Raw prompt vs Chat template — what's the difference

Raw prompt: feeds your text directly to the model for continuation. The model has no idea who's speaking or who's asking whom (same as Tab ①).

Product layer (chat template): wraps your text with Qwen3's agreed markers before sending. The markers look like this:

<|im_start|>system
You are X
<|im_end|>
<|im_start|>user
help me
<|im_end|>

This tells the model "this part is system, this part is user". Expand the "Final prompt actually sent to the model" preview below to see exactly how the markers wrap it.

(?) But what IS <|im_start|> exactly?

It's not a string — it's a single token.

Looks like 12 characters, but to the model it's 1 token (Qwen3 vocab id 151644). The tokenizer recognises it atomically — won't split into </|/i/m/...

"<|im_start|>"  ← what you see (12 chars)
[151644]        ← what the model sees (1 token id)

Lives in the same vocab list as 一 / 年 / 月 — entry #151644.

Why does this token exist?

The model essentially only does "continue the token sequence" — give it text, it predicts the next token.

Problem: feed it 一年有幾個月? — how does it know to answer, vs essay-continue (like "一年有幾個月? 答案是...")?

<|im_start|> / <|im_end|> are role-boundary markers:

<|im_start|>user        ← "user starts speaking"
一年有幾個月?
<|im_end|>              ← "user is done"

<|im_start|>assistant   ← "now you (assistant) speak"

During training, the model saw billions of conversations in this format and learned: "after <|im_start|>assistant\n, generate a reply, stop at <|im_end|>".

        (?) What is Thinking mode / the <think> block
        

          
Reasoning models like Qwen3 can "think in their head" before answering. They emit reasoning tokens wrapped in <think>...</think>, and only then emit the answer.

          
    
            
  • Direct answer (skip thinking): the client force-injects an empty <think></think>, so the model has no room to think → it answers directly. Fast, but easy to get wrong on hard problems.
    • 
            
  • With thinking: leaves room for the model to think → you see it actually write the reasoning out as tokens. Slower, but usually correct.
    • 
          

          
The apple problem (Dad has 3 apples, son has 2 more than him — how many does the son have?) is designed to show this difference — direct answers often miscalculate, thinking gets it right.

        (?) Why does the Agent also have a System Prompt — how is it different from Tab ②
        

          
Same as Tab ② — "speaking to the model up front" — but this time the content includes "you have these tools available: get_time / read_file / write_file / exec_bash" plus a JSON schema describing each tool.

          
Once the model sees this, when it hits a question that needs a tool (e.g. "What time is it?") it emits a <tool_call> tag to invoke the matching tool.

        (?) What are Agent / Tool / <tool_call> / Turn
        

          

            
Agent: a chatbot answers what you ask. An agent calls tools on its own (read files, write files, run shell...) to get things done, then answers. Underneath it's still token prediction — just with a trigger pattern the client knows how to recognize.

          

          

            
Tool: a ready-made Python function (this repo ships 4: get_time / read_file / write_file / exec_bash) plus a JSON schema describing it.

          

          

            
<tool_call> ... </tool_call>: Qwen3's agreed XML tag. When the model wants to call a tool, it emits this string:

            
<tool_call>
{"name": "get_time", "arguments": {}}
</tool_call>

            
The backend sees it, parses the JSON, calls the matching Python function, then injects the result (e.g. a timestamp) back into the conversation and feeds it to the model again.

          

          

            
Turn: one cycle of the multi-turn loop. Turn 1 = model emits text for the first time (possibly a tool_call). Turn 2 = model sees Turn 1's tool result and emits text again. Repeat until the model stops emitting tool_call → final answer.
