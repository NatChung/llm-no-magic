(?) 什麼是 System prompt

你跟 ChatGPT / Claude 對話時看不到、但前面已被「偷偷填好」的隱性指令。例如:「你是友善客服,只用條列式回答,不超過 3 點」。

SaaS 產品的「加工」就是這層 — 你寫「help me」,產品自動補上指令再送進 model。

這 tab 讓你看見:加 system prompt 跟不加,model 吐的字差多少。

(?) 裸 prompt vs Chat template — 差在哪

裸 prompt(raw):把你打的字直接丟給 model 接龍。Model 不知道誰在講話、誰問誰答(同 Tab ①)。

產品加工(chat template):用 Qwen3 約定的 marker 包好再丟。Marker 長這樣:

<|im_start|>system
你是X
<|im_end|>
<|im_start|>user
help me
<|im_end|>

告訴 model「這段是 system、這段是 user」。展開下面「實際送進 model」preview 就看得到 marker 怎麼包成。

(?) 那 <|im_start|> 到底是什麼?

它們是什麼 — 不是字串,是單一 token

看起來像 12 個字元、model 眼裡是 1 個 token(Qwen3 vocab id 151644)。tokenizer 認識它、不會拆成 </|/i/m/...

"<|im_start|>"  ← 你看到的(12 字元)
[151644]        ← model 看到的(1 個 token id)

跟 一 / 年 / 月 這些一般 token 並列、是 model 詞表內第 151644 號。

為什麼要這個 token

Model 本質只會「接續 token 序列」— 給它一段文字、它預測下一個 token。

問題:給 model 一年有幾個月? 這串文字、它怎麼知道是要 回答、還是 接續(像「一年有幾個月? 答案是...」這樣 essay 風)?

<|im_start|> / <|im_end|> 是角色邊界 marker:

<|im_start|>user        ← 「以下是 user 講的話」
一年有幾個月?
<|im_end|>              ← 「user 講完了」

<|im_start|>assistant   ← 「以下換你(assistant)講」

訓練時 model 看過幾億份這種格式的對話、學會「看到 <|im_start|>assistant\n 後面我該生 reply、生到 <|im_end|> 為止」。

你那段 prompt 完整解讀

<|im_start|>user
一年有幾個月?<|im_end|>
<|im_start|>assistant
<think>

</think>
字面 model 解讀
<|im_start|>user\n「user 開始說話」
一年有幾個月?user 的問題內容
<|im_end|>「user 說完」
<|im_start|>assistant\n「換 assistant 講」
<think>\n\n</think>Qwen3 reasoning marker — 見 Tab ③
(prompt 到此結束)model 從這裡開始生 reply

No magic — 只是 model 訓練時學會的約定。

Agent 跑的軌跡

      
每個 turn 結束都會 累積進 messages,送進下一次 model 看。

      

        (?) 什麼是 Agent / Tool / <tool_call> / Turn
        

          

            
Agent:Chatbot = 你問它答;Agent = 它會自己呼叫工具(讀檔、寫檔、跑 shell...)做事,做完才答。本質還是 token prediction,只是約定 client 認得的 trigger pattern。

          

          

            
Tool:寫好的 Python function(這 repo 有 4 個:get_time / read_file / write_file / exec_bash)+ JSON schema 描述。

          

          

            
<tool_call> ... </tool_call>:Qwen3 約定的 XML 標籤。Model 想呼叫工具時吐這串:

            
<tool_call>
{"name": "get_time", "arguments": {}}
</tool_call>

            
Backend 看到就 parse JSON、call 對應 Python function、結果(時間字串)塞回對話再丟給 model。

          

          

            
Turn:multi-turn loop 一輪。Turn 1 = model 第一次吐字(可能 tool_call)。Turn 2 = model 看 turn 1 tool 結果後再吐字。直到 model 不再 tool_call → final answer。

⑤ 指令、Script、API

        
一章說完「你的電腦怎麼做事」。

        
電腦不是黑魔法。它只懂你怎麼下指令、怎麼把指令串起來、怎麼請別台電腦幫忙做你做不到的事。看懂這三層,後面的 Skill / MCP 自然就懂。

      


      

        
§1. 指令 = 你跟電腦的對話介面

        
電腦有兩種介面:

        
    
          
  • 圖形(GUI):你按 icon、點選單、拖檔案
    • 
          
  • 指令(CLI):你打字告訴電腦做什麼
    • 
        

        
CLI 你電腦裡早就有 — Mac 是 Terminal(在 Applications / Utilities)、Windows 是 PowerShell 或 cmd、Linux 是 shell。打開後是黑底白字、閃爍的游標等你打字。

        
電腦預設就懂一堆指令(作業系統 macOS / Linux / Windows 出廠內建):

        
# 預設指令(出廠就有)
ls                    # 列出這個資料夾有什麼
cd Downloads          # 換到 Downloads 資料夾
mkdir photos          # 開一個叫 photos 的新資料夾
mv *.jpg photos/      # 把所有 .jpg 搬到 photos/

        
這些是「電腦的母語單字」。但內建的有限,要做特殊事(壓縮影片、抓網路資料、處理 PDF)就要 裝擴充:

        
# 裝擴充能力
brew install ffmpeg   # macOS:裝影片處理工具
brew install yt-dlp   # 裝 YouTube 下載工具
brew install jq       # 裝 JSON 處理工具

        
裝完後,電腦多懂幾個「單字」。ffmpeg 就能用了。

        

          白話: 指令 = 電腦的詞彙。預設一份,可以擴充。
        

      


      

        
§2. Script = 把指令結合起來

        
單條指令做一件事。實際任務常常 很多步。

        
舉例「整理 ~/Downloads」:

        
    
          
  1. 列出所有檔案
    2. 
          
  2. 判斷 每個檔案的副檔名
    3. 
          
  3. 對 .pdf:搬到 pdfs/
    4. 
          
  4. 對 .png / .jpg:搬到 images/
    5. 
          
  5. 對其他:搬到 misc/
    6. 
        

        
這需要兩件事:

        
    
          
  • 重複(每個檔案都跑一次)
    • 
          
  • 判斷(看副檔名決定搬去哪)
    • 
        

        
把多條指令 打包成一個檔案、加上 判斷跟迴圈,就叫 script。

        
#!/bin/bash
# organize.sh — 整理 ~/Downloads

for file in ~/Downloads/*; do
  case "${file##*.}" in
    pdf)        mv "$file" ~/Downloads/pdfs/ ;;
    png|jpg)    mv "$file" ~/Downloads/images/ ;;
    *)          mv "$file" ~/Downloads/misc/ ;;
  esac
done

        
    
          
  • for = 重複(每個檔案執行一次)
    • 
          
  • case = 判斷(看副檔名決定)
    • 
          
  • mv = 內建指令
    • 
        

        
語法細節看不懂沒關係 — 看出有「for(重複)+ case(判斷)」就達到本段目的:script = 一串指令 + 控制流。

        
「重複 + 判斷 + 順序」三件事合稱 控制流(control flow)。Script 強過單條指令的地方就在這。

        
存成 organize.sh、跑 bash organize.sh — 電腦自動整理 30 個檔案、你喝口水。

        

          
白話: Script = 一串指令加判斷跟迴圈,存成檔案、按一下就跑完。

          
2026 實話: 你大概不會自己 Google bash 語法,實務上會請 AI 幫你寫。但你要看得懂這層 — 不然 AI 寫錯你不知道。

        

      


      

        
§3. API = 雇用另一台電腦

        
問題:你的電腦不是無所不能。

        
比喻時間。真實世界裡你也不是無所不能 — 牙痛你不會自己拔牙,你找牙醫。為什麼?

        
    
          
  • 牙醫腦袋裡有不同 知識(咬合、神經分布)
    • 
          
  • 手上有不同 技能(鑽頭、麻醉)
    • 
        

        
電腦世界一樣。每台電腦都有不同知識 + 不同技能。例如:

        

          
            

              電腦
              腦袋裡裝什麼(知識)
              手上能做什麼(技能)
            

          
          
            
資料庫主機公司 10 年訂單資料極快查詢

            
氣象主機連著感測網預報天氣

            
支付主機(Stripe)金融法規、各國支付規格刷卡扣款

            
AI 主機訓練好的大型模型文字生成、翻譯

          
        

        
你的筆電做不到這些 — 資料太大、設備太特殊、要金融牌照、要超大運算。

        
那怎麼辦?跟真實世界一樣的答案:請別人。

        
電腦請別的電腦的方式叫 API(Application Programming Interface):

        
# 問天氣主機:台北現在天氣?
curl "https://api.weatherapi.com/v1/current.json?key=YOUR_KEY&q=Taipei"

# 回:
{"current":{"temp_c": 28, "condition":{"text": "Sunny"}}}

        
curl 是內建指令、發出網路請求。天氣主機收到、查它的感測網、回個 JSON 給你。

        
幾乎所有雲端服務(地圖、翻譯、搜尋、支付、寄信)都有 API 介面。它本來就是 給程式呼叫用的 — 所以名字裡才有 "Programming"。

        

          白話: API = 你的電腦透過網路請別台電腦幫忙做事的標準格式。

⑦ MCP

        
工具不固定的時候、AI 自己選工具。

      


      

        
§1. 回顧:你還缺一塊

        
到 ⑥ 章你會了:

        
    
          
  • 指令 / script / API(⑤) — 知道怎麼讓電腦做事
    • 
          
  • Skill(⑥) — 知道怎麼把上面包成可選的能力,AI 按需載入
    • 
        

        
但這套有個前提:工具是固定的、事先決定好的。你寫好 weather.py 在 skill 裡,AI 就只能用它查天氣。

        
問題來了:工具不固定怎麼辦?

      


      

        
§2. 主場景:操控網頁

        
舉例「幫我從蝦皮抓某商品的價格」。

        
你寫 script?Amazon 結構跟蝦皮不同,腳本只能對一個網站。

        
寫 API?有些網站有公開 API、有些沒。寫不完。

        
更糟:網站隨時改版。你今天能 work 的 script,下週可能整個壞。

        
那 AI 怎麼做?

        
直接「看著辦」:讓 AI 看當前網頁長啥樣 → 自己決定點哪個按鈕、抓哪段文字。

        
但 AI 自己看不到網頁,要有人幫它「看 + 操作」。這就是 MCP 要解的問題。

      


      

        
§3. MCP 是什麼

        
MCP = Model Context Protocol。Anthropic 2024 年 11 月公開、開放規格,所有人都能實作。

        
解決的問題:AI 跟工具的固定 binding 太死板。

        
比喻:USB-C for AI applications(Anthropic 自家用詞)。

        

          
            

              場景
              過去
              現在
            

          
          
            
充電線各家專屬充電頭 / mini-USB / micro-USB / Lightning...USB-C 一條線、所有手機平板都能用

            
AI 接工具每家 AI 自己寫整合(Claude 一套、ChatGPT 一套)一個 MCP protocol、任何 AI 接任何工具

          
        

        
運作模式:client(AI 那邊)跟 server(工具那邊)透過 JSON-RPC 對話,client 動態問 server 有什麼工具、按需呼叫。完整 trace 見 §4。

        
關鍵點:工具不是事先寫死的、是執行時動態問出來。換 server 就換工具集、不用改 AI 那邊一行 code。

      


      

        
§4. 實際 trace 範例

        
用 Playwright MCP server(專門讓 AI 控瀏覽器的 server,Microsoft 開源)看一個簡單任務。

        
任務:「幫我截圖 example.com」。

        
→ tools/list                            ← AI 啟動,問 server 你能做什麼?
← [browser_navigate, browser_click, browser_screenshot, ...]
                                        ← server 回它能做的事

→ browser_navigate(url="example.com")   ← AI 挑「打開網頁」
← {success: true, title: "Example Domain"}
                                        ← server 真的開了瀏覽器

→ browser_screenshot()                  ← AI 挑「截圖」
← {image: "data:image/png;base64,..."}  ← 回截圖 binary

→ "已截圖 example.com,結果如上"        ← AI 回完整對話

        
幾個觀察:

        
    
          
  • AI 完全不需要事先知道 server 有什麼工具 — 啟動才問
    • 
          
  • 工具清單可以換 server 就換 — 想控資料庫?換 DB MCP server,工具變成 db_query / db_insert 之類
    • 
          
  • AI 那邊的 code 不變,只是 talk 的對象換了
    • 
          
  • 裝 MCP server 等於給它對應的權限 — Playwright MCP 能控你瀏覽器、Filesystem MCP 能讀你檔案。用之前看清楚它能做什麼、信不信任(社群維護的 server 跟自家寫的 server 信任程度不一樣)
    • 
        

        
這就是「一個 protocol、所有工具」的價值。
